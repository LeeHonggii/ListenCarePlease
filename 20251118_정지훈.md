# To-Do(향후 일정)
## 생각해야할 내용
1. LLM 없이 가능한가
2. 회의록에 있는 내용만 To-Do로 구현을 할건가
3. LLM으로 간다면 어떤 방식으로 갈건가

회의록 전사 텍스트(STT 결과물)를 **'실행 가능한 할 일(Actionable To-Do)'**로 변환하는 과정은 크게 4단계의 파이프라인으로 나눌 수 있습니다.
구체적으로 어떤 과정을 거쳐야 개발할 수 있는지, 기술적인 흐름을 단계별로 설명해 드릴게요.

** 1단계는 완료 **
[개발 프로세스] 회의록을 To-Do로 바꾸는 4단계 파이프라인
1단계: 전처리 및 화자 분리 (Preprocessing & Diarization)
STT로 변환된 텍스트만으로는 부족합니다. "누가" 그 말을 했는지가 가장 중요합니다.
핵심 작업: 화자 분리(Speaker Diarization) 기술이 선행되어야 합니다.
이유: "김 대리, 내일까지 보고서 보내줘요"라는 문장에서 '보낼 사람(Assignee)'이 김 대리라는 것을 알기 위해서입니다.
데이터 형태 예시:
(User A): 다음 주 월요일까지 기획안 초안 작성해서 공유해 주세요. (User B): 네, 알겠습니다. 제가 준비하겠습니다.

2단계: 의도 파악 및 추출 (LLM Extraction)
이제 LLM(GPT-4o, Claude 3.5 Sonnet 등)에게 텍스트를 던져주고 To-Do를 뽑아내라고 시키는 단계입니다. 이때 프롬프트 엔지니어링이 핵심입니다.
핵심 로직: 단순히 문장을 요약하는 게 아니라, 조건부 추출을 해야 합니다.
명령문/제안문 탐지: "~해주세요", "~하겠습니다", "~합시다" 같은 종결 어미를 찾습니다.
수락 여부 확인: A가 시켰는데 B가 "그건 힘들 것 같습니다"라고 거절했다면 To-Do가 아닙니다. **문맥(Context)**을 보고 합의된 사항만 뽑아야 합니다.
프롬프트 예시:
"다음 회의록에서 '합의된 업무(Agreed Tasks)'만 JSON 형식으로 추출해. 거절되거나 보류된 안건은 제외해."

3단계: 데이터 구조화 (Structuring)
LLM이 내뱉은 자연어를 시스템이 이해할 수 있는 **데이터(JSON)**로 바꿉니다. 사용자님이 관심 가지셨던 **NER(개체명 인식)**이 여기서도 쓰입니다.
추출할 필드:
Task: 업무 내용 (예: 기획안 초안 작성)
Assignee: 담당자 (예: 김 대리 -> User B)
Due Date: 마감기한 (예: 다음 주 월요일 -> 2025-11-24)
Priority: 중요도 (문맥상 급하다면 High, 아니면 Medium)
💡 팁: 날짜 변환 (Relative Date Resolution) 회의에서는 "다음 주 화요일", "모레까지"라고 말합니다. 이를 실제 날짜(YYYY-MM-DD)로 변환해 주는 로직을 프롬프트에 포함하거나 별도 함수로 처리해야 합니다.

4단계: 연동 및 등록 (Integration via MCP/API)
마지막으로 구조화된 데이터를 실제 업무 툴(Jira, Notion, Slack 등)에 등록합니다.
여기서 MCP가 활약합니다:
이전 질문에서 다뤘던 **MCP(Model Context Protocol)**를 사용하면, LLM이 추출한 To-Do JSON 데이터를 **내 노션(Notion)이나 지라(Jira)**에 직접 API 코드를 짜지 않고도 연결해서 꽂아 넣을 수 있습니다.
실제 데이터 처리 예시
[입력: STT 회의록]
팀장: 자, 이번 UI 디자인 시안 수정 건은 누가 맡을까요? 민수: 제가 하겠습니다. 이번 주 금요일까지 1차 수정본 올릴게요. 팀장: 좋아요. 그리고 영희 씨는 경쟁사 분석 자료 좀 찾아봐 줄래요? 영희: 아, 팀장님. 제가 지금 다른 프로젝트 마감이 겹쳐서 그건 다음 주에 가능할 것 같은데요. 팀장: 그럼 급한 건 아니니 다음 주 수요일까지 해주세요.

⬇️ [LLM 처리 결과 (JSON)]

JSON
[
  {
    "task": "UI 디자인 시안 1차 수정본 업로드",
    "assignee": "민수",
    "due_date": "2025-11-21",  // '이번 주 금요일' 변환
    "status": "To Do",
    "priority": "High"
  },
  {
    "task": "경쟁사 분석 자료 조사",
    "assignee": "영희",
    "due_date": "2025-11-26",  // '다음 주 수요일' 변환
    "status": "To Do",
    "priority": "Low"  // '급한 건 아니니'라는 문맥 반영
  }
]
개발 시 고려해야 할 난관 (Tip)
모호한 화법: "이거 한번 검토해 보는 게 어때요?"는 진짜 하라는 건지, 그냥 의견인지 모호할 때가 많습니다. 프롬프트에 **"명확한 실행 의사가 있는 경우만 추출하라"**는 지침을 줘야 합니다.

주어 생략: 한국어는 주어를 자주 생략합니다. "그럼 내일까지 보낼게요"라고 했을 때 누가 보내는지 앞 문맥을 보고 추론해야 하므로, 이전 대화 이력(History)을 충분히 제공해야 합니다.

혹시 이 과정을 직접 테스트해 보실 수 있도록, 간단한 프롬프트 템플릿을 하나 짜드릴까요? 아니면 JSON으로 날짜를 변환하는 로직 쪽이 더 궁금하신가요?
